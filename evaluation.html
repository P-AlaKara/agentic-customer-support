Abstract

Abstract

This report presents a summative evaluation of Rehema, a multi-agent, session-based customer support system designed for e-commerce operations. The evaluation was conducted against a live locally hosted instance of the system and assessed four primary performance metrics: Intent Recognition Accuracy, Task Success Rate, Escalation Rate, and Average Resolution Time. A fifth metric — Response Quality via LLM-as-Judge — was planned but deferred owing to an absent API key configuration.

Per-case analysis of a 30-utterance intent benchmark yields a derived accuracy of 90.0% (27/30). All eight multi-turn conversation flows produced outcomes consistent with their expected design, representing a 100% outcome-prediction accuracy. Six of eight flows (75%) were resolved autonomously; the remaining two (25%) were correctly escalated to human agents. Historical analytics drawn from the live database (n = 20 conversations) corroborate these findings, showing a 70% autonomous resolution rate and a 97.1% high-confidence intent classification rate. Discrepancies between aggregate summary statistics and per-case detail records in the evaluation artefact are documented and attributed to a session-ID matching failure in the event-log polling routine; per-case details are used as the authoritative data source throughout.

1. Introduction

1.1 Background and Motivation

Rehema is a multi-agent, session-based customer support system designed to automate routine e-commerce customer service interactions. The system is activated by a natural-language wake phrase (hey rehema) and maintains conversational context using unique session identifiers. Upon activation, it classifies the user's intent, routes the request to an appropriate specialised sub-agent, and either resolves the issue autonomously or escalates to a human operator when the request falls outside its scope. The system's API gateway is implemented using FastAPI and exposes administrative endpoints for analytics, agent event logs, and health checks.

As the system approaches operational readiness, a rigorous summative evaluation is necessary to characterise its performance, quantify its failure modes, and establish an evidence-based baseline for iterative improvement. This document reports the results of such an evaluation, conducted on 15 May 2026 against a locally deployed instance of the system.

1.2 Evaluation Objectives

The evaluation sought to answer the following research questions:

  1. How accurately does the system classify user intent across a representative benchmark of labelled utterances, and are there systematic patterns in misclassifications?
  2. To what degree does the system successfully resolve customer requests autonomously end-to-end?
  3. Is the system's escalation logic consistent and appropriate — escalating when required and resolving when able?
  4. What end-to-end latency does the system exhibit under realistic evaluation conditions?
  5. (Planned) How is the quality of natural-language responses rated by an independent AI judge across dimensions of relevance, correctness, and naturalness?

2. System Overview

2.1 Architecture

Rehema follows a coordinator–agent architecture. A FastAPI-based gateway receives HTTP POST requests at /chat, forwards them to a central coordinator, and the coordinator invokes an intent recognition agent before dispatching to the appropriate specialised agent. If the classified intent maps to an unsupported task, or if the user explicitly requests human assistance, a human-handoff signal is issued. The architecture is illustrated in Figure 1.

3.2 Test Dataset

3.2.1 Intent Recognition Benchmark (30 utterances)

A ground-truth dataset of 30 utterances was constructed spanning all 7 supported intent classes. The dataset deliberately includes both direct, canonical phrasings (e.g., "I want to return this item") and indirect, colloquial, or synonym-based expressions (e.g., "I need to send this back", "Can I exchange this product?") to probe the robustness of the intent recogniser beyond its likely training vocabulary. Class distribution: track_order (7), process_return (6), general_inquiry (4), onboarding (4), greeting (3), close_conversation (3), request_human (3).

3.2.2 Multi-Turn Conversation Flows (8 scenarios)

Eight multi-turn conversation scenarios were designed to test end-to-end system behaviour. Each begins with the wake phrase hey rehema followed by one or two additional messages. Scenarios cover happy-path resolution for the four main task types, one general inquiry, one greeting-only interaction, one multi-turn tracking-then-close flow, and two intended escalation scenarios (account issue requiring back-end privileged access; explicit human agent request). Each scenario carries an annotated expected outcome of either resolved or escalated.

3.2.3 Response Quality Test Cases (6 cases — deferred)

Six test cases were prepared for LLM-as-Judge evaluation covering: wake/greeting, return request, order tracking, onboarding, general policy inquiry, and shipping information inquiry. These were not executed in the current run.

3.4 Evaluation Procedure

For each intent benchmark utterance, the harness first activated a fresh session via hey rehema, then dispatched the test utterance on that session. It then polled /admin/agent/intent/events, matching by session ID and event direction (subscribed), to retrieve the classified intent and confidence score. A 20-second sleep was enforced between trials to respect upstream API rate limits.

For each conversation flow, messages were sent sequentially on a shared session. After each non-wake message, the response was inspected for the is_human_handoff flag or an escalated status value; a positive result immediately terminated the flow as escalated. Wall-clock time was measured from the first message dispatch to the receipt of the final response.

4.4 Resolution Time Analysis

Wall-clock resolution times across the eight flows span 9.698s to 67.487s. These figures are substantially influenced by the mandatory 20-second inter-message sleep enforced by the evaluation harness to respect API rate limits, and should not be interpreted as indicative of production end-to-end latency. The following structural observations apply:

  • Single-turn (Greeting Only, 22.5s) reflects one API call plus the 20-second post-wake-phrase sleep mandated before the inter-trial delay. This provides the clearest estimate of baseline API latency: approximately 2–3 seconds of actual processing per call.
  • Two-turn resolved flows (44.6s – 66.3s) each involve two API calls and one 20-second intra-flow sleep, yielding approximately 25–47 seconds of actual inference and overhead time.
  • Three-turn flow (Multi-Turn: Track then Close, 67.5s) involves three API calls and two 20-second sleeps, consistent with the observed time.
  • Manual escalation (25.0s) confirms the expected 2-call pattern with one 20-second sleep prior to the escalation-triggering message.
  • Account Issue escalation anomaly (9.698s) is unexpectedly fast — shorter than the 20-second inter-message sleep alone. This strongly suggests the escalation signal was returned on the initial hey rehema wake message rather than on the subsequent task message, possibly due to residual session state from a preceding flow. This behaviour warrants investigation, as the wake phrase should not independently trigger escalation.

Both escalation paths functioned correctly in the benchmark (100% outcome-prediction accuracy), and the historical rate of 30% suggests that approximately one in three real-world conversations involves a request the system appropriately defers — a reasonable threshold for a first-generation deployment.

5.1 LLM-as-a-Judge: Concept and Rationale

Traditional automated NLP metrics — BLEU, ROUGE, METEOR, and their variants — were designed primarily for reference-based translation and summarisation tasks, where a small set of human-authored reference outputs exists. In open-domain dialogue and customer support applications, no single "correct" response exists for a given query; responses may be correct, relevant, and natural in many different phrasings. Reference-based metrics therefore correlate poorly with human judgements of response quality in these settings.

LLM-as-a-Judge addresses this gap by leveraging a capable large language model — configured with a specific evaluation persona, domain context, and structured scoring rubric — to assign quality scores to candidate responses on multiple dimensions. The judge model reads the user message and the bot response (and optionally a context hint), then returns structured scores in a format such as JSON. Key advantages of this approach include:

  • Rubric flexibility: The judge can be instructed to evaluate any domain-specific quality criterion, including e-commerce policy correctness, empathy, and appropriate use of escalation language.
  • Scalability: Thousands of response pairs can be evaluated at a fraction of the cost and time of human annotation panels.
  • Consistency: With temperature=0.0, the judge produces deterministic, reproducible scores across evaluation runs.
  • Multi-dimensional assessment: Unlike single-figure metrics, LLM judges simultaneously score independent quality dimensions, enabling fine-grained diagnostic insight.

5.2 Planned Evaluation Criteria and Rubric

The evaluation script defines three scoring dimensions, each rated on a 1–5 integer scale:

DimensionScore 5 — ExcellentScore 1 — Poor
Relevance Response directly and completely addresses the user's stated need or intent Response is off-topic or addresses a fundamentally different intent
Correctness All factual, procedural, and policy content is accurate for an e-commerce support context Contains factual errors, invalid instructions, or incorrect policy information
Naturalness Response is warm, fluent, and human-like in tone, phrasing, and register Response is robotic, repetitive, awkward, or inappropriately formal / informal

The planned judge model is Gemini 2.0 Flash, configured with temperature=0.0 and max_output_tokens=120 to enforce deterministic, concise output. The judge is prompted to return scores as a single JSON line with an optional brief rationale comment (≤12 words).

5.3 Risks and Limitations of LLM-as-a-Judge

  • Same-family bias: Using a Gemini model to judge a Gemini-backed system may introduce systematic bias in favour of responses stylistically aligned with the judge's generative preferences. Cross-model judging (e.g., GPT-4o judging Gemini outputs) is recommended as a supplementary validation.
  • Prompt sensitivity: Small changes to the judge prompt — including persona wording, rubric phrasing, or output format instructions — can shift scores non-trivially. The rubric should be pilot-tested and validated before large-scale deployment.
  • Absence of ground-truth human annotations: Without human-labelled reference judgements, it is difficult to assess whether the judge's scores are well-calibrated relative to actual user satisfaction.
  • Truncated context: The judge receives only the most recent user message and bot response; prior conversational turns are omitted. This may lead to incorrect relevance judgements for responses that are only coherent in context.
  • Reproducibility at scale: While temperature=0.0 is specified, minor sampling non-determinism at token probability boundaries may produce slightly different scores across runs; averaging over multiple passes is recommended for high-stakes comparisons.

6. Discussion and Interpretation

6.1 Intent Recognition

A 90.0% intent recognition accuracy on a 30-utterance benchmark is a reasonable result for a first-generation system, especially given the dataset's intentional inclusion of indirect and synonym-bearing phrasings. The perfect accuracy on five of seven classes demonstrates that the recogniser handles direct, explicit formulations reliably. The pattern of failures, however, is highly informative: all three misclassifications share the characteristic of being indirect, idiomatic, or action-verb-framed expressions that paraphrase a supported intent without using any of its canonical keywords.

Specifically, "I need to send this back" and "Can I exchange this product?" are semantically equivalent to "I want to return this item", but use the colloquial verb "send back" and the domain synonym "exchange" respectively — neither of which appears to be represented in the classifier's pattern or embedding space. Similarly, "Connect me to an agent please" uses an instruction verb ("connect") rather than the first-person declarative form ("I want to speak to...") that the classifier correctly handles. These failures suggest the intent recogniser relies on relatively narrow pattern matching or a constrained embedding model, rather than robust semantic generalisation over paraphrase space. Augmenting the training set with paraphrase variants, or adopting a more powerful LLM-based embedding classifier, would likely eliminate all three failure cases.

6.2 Conversation Flow Outcomes and Escalation

The 100% outcome-prediction accuracy across all eight flows is the evaluation's most significant finding. It validates that the coordinator's routing and escalation logic functions exactly as designed under the tested conditions. The 75% autonomous resolution rate (6/8 flows) is not a performance deficit — it reflects the intentional architecture decision to escalate when the system lacks the capability or authorisation to resolve an issue autonomously. The two escalation flows were correctly identified and appropriately routed to a human handoff in both cases.

The historical escalation rate of 30% (vs. 25% in the benchmark) is consistent with the natural long-tail of user requests in real-world deployments. Users in production may request account-level changes, dispute resolution, or complex multi-step issues that the system is not yet equipped to handle. The 5-percentage-point gap between benchmark and historical escalation rates is small enough to suggest the benchmark is a reasonable proxy for live traffic.

6.3 Resolution Time and Latency

As discussed in Section 4.4, the observed resolution times are dominated by the evaluation harness's mandatory 20-second inter-message delay. This inflates all multi-turn flow times by 20–40 seconds relative to what a production deployment without rate limiting would observe. The Greeting Only flow (22.5s, 1 API call) provides the most useful latency estimate: discounting the 20-second post-wake delay, actual API response time is approximately 2–3 seconds per call under the current hardware constraints. In production, with a dedicated server and no rate-limit sleeping, multi-turn conversations would plausibly complete in under 15 seconds for two-turn flows.

The Account Issue escalation anomaly (9.698s) is noteworthy. Given that this duration is shorter than a single 20-second sleep, it is almost certain that only one API call was made — meaning the wake phrase hey rehema itself returned an escalated status. This is an unexpected and undesirable behaviour that warrants a targeted debugging session.

6.4 Historical Analytics Cross-Validation

The strong agreement between benchmark results and historical analytics (resolution rates of 75% and 70%; escalation rates of 25% and 30%) provides informal cross-validation that the benchmark test set is ecologically representative. The historical 97.1% high-confidence intent rate is particularly encouraging: it implies that in the vast majority of real-world interactions, the system classifies intent with high certainty, and only abstains (returning unknown) when genuinely unsure — a safe and transparent failure mode.

7. Limitations

  1. Small benchmark size: 30 utterances and 8 flows are insufficient for high-confidence statistical inference. Reported accuracy figures carry wide confidence intervals and should not be treated as definitive performance certificates.
  2. Single evaluation run: All results derive from one execution. Run-to-run variability due to LLM sampling non-determinism, system load, and network conditions was not characterised.
  3. Rate-limit-inflated latencies: The 20-second inter-call delay dominates all resolution time figures, making latency comparisons across scenarios unreliable as production estimates.
  4. Co-located system and evaluator: Sharing 8 GB RAM between the LLM-backed server and the Python harness introduces potential resource contention that may inflate measured response times.
  5. Response quality not assessed: Metric 5 (LLM-as-Judge) was skipped in its entirety. There is currently no evidence about the linguistic quality, tone, or correctness of the system's natural-language responses.
  6. Instrumentation discrepancies: The aggregate summary statistics in report.json contain known errors. Although per-case details appear accurate, the root cause was not formally diagnosed and may indicate broader reliability issues in the evaluation harness.
  7. No adversarial or robustness testing: The benchmark contains no adversarial inputs, offensive language, multi-intent queries, or linguistically ambiguous utterances. System robustness in edge cases is unknown.
  8. Synthetic evaluation identity: All trials used eval@test.com as the customer email. Any customer-specific personalisation, loyalty-tier logic, or order-history-dependent behaviours were not exercised.
  9. Account Issue anomaly unresolved: The unexpectedly fast resolution (9.698s) of the Account Issue escalation flow suggests a potential session-state bug that was not investigated in this evaluation.

8. Conclusions

This evaluation provides a foundational performance characterisation of the Rehema agentic customer support system. The principal findings are summarised below:

  • The intent recogniser correctly classified 27 of 30 benchmark utterances (90.0%), with all misclassifications attributable to indirect or idiomatic phrasings that fall outside the classifier's current vocabulary — not to fundamental intent confusion.
  • All eight multi-turn conversation flows produced correct outcomes (100% outcome-prediction accuracy), validating both the routing logic for resolution scenarios and the escalation policies for unsupported tasks and explicit human requests.
  • The system autonomously resolved 75% of evaluation flows and 70% of historical conversations, with all escalations occurring appropriately and by design.
  • The historical high-confidence intent rate of 97.1% confirms that in real-world usage the system is almost universally decisive in its classifications, reserving abstention for genuinely ambiguous inputs.
  • Response quality evaluation is the highest-priority outstanding item for the next evaluation cycle, as no evidence currently exists regarding the linguistic quality of the system's outputs.

The most actionable near-term improvement identified by this evaluation is the augmentation of the intent recognition vocabulary for the process_return and request_human classes to cover indirect phrasings, action-verb forms, and domain synonyms. Based on the three observed failure cases, targeted additions of approximately 5–10 paraphrase variants per affected class would plausibly close the accuracy gap entirely on the current benchmark.

Secondary priorities include: (1) investigating and resolving the Account Issue escalation anomaly (potential session-state bleed-across); (2) executing the deferred LLM-as-Judge response quality evaluation; (3) expanding the benchmark to at least 100 utterances and 20 conversation flows to enable statistically meaningful confidence intervals; and (4) refactoring the evaluation script's event-log polling and aggregation logic to eliminate the instrumentation discrepancy identified in this run.

Overall, the evaluation results support the conclusion that Rehema is functionally viable and behaviourally consistent as a first-generation automated customer support agent. Its intent recognition and escalation behaviour are demonstrably correct for the tested scenarios, and historical analytics corroborate the benchmark findings. With targeted vocabulary improvements and a completed response quality evaluation, the system will be well-positioned for broader deployment validation.
